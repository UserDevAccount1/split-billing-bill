How it works

A short manual: what the app does, how the fair-split math works, the rules for bad input, the API, and how to run it in one container.

What Split does

Split answers one question precisely: “We owe a total of X — who pays how much?”. You enter a total and the people involved. Anyone can be given a larger share (a weight) when they should cover more — for example a share of 2 means that person pays twice what a share of 1 person pays. Split then shows each person's amount, rounded to the cent, and guarantees the amounts add back up to the exact total.

Using the dashboard

1
Name the bill (optional) — e.g. “Friday dinner”. It's just a label for the saved record.
2
Enter the total and pick a currency. The receipt on the right updates live as you type.
3
Add people and set each person's share. Leave every share at 1 for an even split; bump one to 2 (or more) for an uneven one.
4
Watch the stamp. When the per-person amounts reconcile to the exact total, the receipt shows a green “✓ reconciles exactly” stamp.
5
Save to store it. Saved splits appear below the form; each can be re-opened to edit or delete — that's the full create / read / update / delete cycle.

The fair-split math

Splitting money is deceptively tricky. If you split $100.00 three ways, each “true” share is $33.333…. Round each to $33.33 and you've only handed out $99.99 — a penny vanishes. Round up and you hand out $100.02 — two extra pennies appear. Neither is acceptable for money.

Split solves this with the largest-remainder method (also called Hamilton's method), working entirely in whole cents so floating-point drift can't creep in:

  1. Convert the total to an integer number of cents.
  2. Give each person their exact fractional share of those cents, then round each down (floor).
  3. Flooring leaves a few leftover cents (always fewer than the number of people).
  4. Hand those leftover cents out one at a time to the people with the largest fractional parts — the ones who were rounded down the most.
The guarantee: because every leftover cent is distributed rather than dropped, the per-person amounts always sum to the exact total — no missing or extra cents, for any total, any number of people, and any mix of shares. This property is locked in by an automated test that checks 2,000 randomized splits on every run.

Worked example — $100.00, evenly, 3 people

Total = 10000¢. Each exact share = 3333.33…¢. Floor each → 3333, 3333, 3333 = 9999¢. One cent is left over; it goes to the largest fractional part (the first person). Result: $33.34, $33.33, $33.33 — which sums to exactly $100.00.

Worked example — $100.00, shares 2 · 1 · 1

Total weight = 4. Shares split the 10000¢ as 5000, 2500, 2500 → $50.00, $25.00, $25.00.

Handling bad input

Split validates before it computes and shows a clear message instead of a wrong number or a crash:

  • Empty or non-numeric total → “Total must be a valid number.”
  • Zero or negative total → “Total must be greater than 0.”
  • No people → “Add at least one person to split between.”
  • A share that is zero, negative, or not a number → “Weight for person N must be greater than 0” (or “…a valid number”).
  • A missing share defaults to 1. A total with extra precision is rounded to the nearest cent.

The same validation runs in the browser (for instant feedback) and again on the server (as the authoritative gate), so a bad value can never reach the database.

REST API

The dashboard is a thin client over a small JSON API. A saved bill stores its inputs and the computed per-person amounts.

MethodPathPurpose
GET/api/billsList all saved splits (newest first).
GET/api/bills/:idRead one split.
POST/api/billsValidate, compute, and create a split.
PUT/api/bills/:idUpdate a split (recomputes the amounts).
DELETE/api/bills/:idDelete a split.
GET/api/healthLiveness + DB status (drives the Status page).

Create a split from the command line:

curl -X POST http://localhost:3000/api/bills \
  -H "Content-Type: application/json" \
  -d '{"name":"Dinner","total":100,
       "participants":[{"name":"Ana","weight":2},
                       {"name":"Bob"},{"name":"Cy"}]}'

Running it — one container

The entire app — the Node/Express server, the vanilla-JS frontend, and the SQLite database — ships inside a single Docker image. There is no external database to connect to.

# build the image
docker build -t split-billing-bill .

# run the one container
docker run -p 3000:3000 split-billing-bill

# then open http://localhost:3000
Self-contained: the SQLite file lives at /app/data/bills.db inside the container. To keep saved splits across restarts, mount a volume: docker run -p 3000:3000 -v split-data:/app/data split-billing-bill.

Running without Docker

npm install
npm start        # serves http://localhost:3000
npm test         # runs the split-algorithm test suite

Requires Node 22.5+ (for the built-in node:sqlite module — no native build step).
